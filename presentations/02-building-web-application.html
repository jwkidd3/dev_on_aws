Building a Web Application on AWS

Monolith vs. Serverless

  • Monolith — one deploy, one DB, one process. Simple at first; painful to scale and change.
  • Serverless — small functions + managed services + pay-per-request
  • You trade server management for service composition. Learning the services is the work.

The App You'll Build

A web app for managing a catalog of items with images.

  • Users sign up and sign in
  • Authenticated users add items with image uploads
  • Each user sees only their items
  • All state in managed AWS services — no EC2 servers

Request Flow

  1. Browser loads static site from S3 / CloudFront
  2. Signs in against Cognito → gets a JWT
  3. Calls an API Gateway endpoint with the JWT
  4. API Gateway authorizes, invokes a Lambda
  5. Lambda reads / writes DynamoDB + S3
  6. Logs and traces land in CloudWatch + X-Ray

Every box is a module in this course.

Why These Services?

  • S3 — cheapest, most durable object storage; static-site hosting in one command
  • DynamoDB — single-digit-ms NoSQL; no servers
  • Lambda — no idle cost; first 1M invocations/month free
  • API Gateway — managed HTTPS, auth, throttling
  • Cognito — sign-up, sign-in, MFA, JWTs without building an identity service

Where You'd Pick Differently

Serverless isn't always the answer.

  • Compute > 15 min → ECS/Fargate or EC2
  • Complex joins / ad-hoc queries → Aurora or RDS
  • High-throughput streaming → Kinesis, MSK
  • Strict p99 latency in a single region → containers
