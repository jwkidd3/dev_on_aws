Getting Started with Permissions

IAM for developers — identities, policies, roles.

The Four Building Blocks

  • User — long-lived identity with credentials (usually a human)
  • Group — bag of users that share a policy attachment
  • Role — temporary identity assumed by a service or another account
  • Policy — JSON document listing what's allowed or denied

Anatomy of a Policy

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect":   "Allow",
    "Action":   ["s3:GetObject", "s3:PutObject"],
    "Resource": "arn:aws:s3:::my-app-uploads/*"
  }]
}

Effect · Action · Resource · optional Condition.

How IAM Decides

  1. Start with implicit deny
  2. Any explicit Deny matches → deny. Stop.
  3. Any explicit Allow matches → allow.
  4. Otherwise → deny.

Explicit deny beats everything. Use sparingly but deliberately.

Roles for Code

  • Code on AWS (Lambda, EC2, ECS) should never have static keys
  • Attach a role — SDK auto-picks up rotating temp credentials
  • Every role has two policies: permissions (what it can do) + trust (who can assume it)

Least Privilege

  • Grant the narrowest actions on the narrowest resources
  • Start with an AWS-managed policy if it fits, then trim
  • IAM Access Analyzer generates a policy from real CloudTrail usage
  • Access Advisor shows last-used dates per service

Debugging "Access Denied"

  1. Read the full error — newer messages name the policy
  2. aws sts get-caller-identity — confirm which identity
  3. Check the region — wrong region looks like missing permission
  4. Check resource-based policies (bucket / KMS / queue) too — they can deny
  5. Last resort: IAM Policy Simulator with the exact action and ARN
