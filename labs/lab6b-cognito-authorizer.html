🎓 Lab 6b

API Gateway Authorizer & Swagger Import

Module 12 — Access 45 min

Objectives 3 min

  • Attach a Cognito user pool authorizer to the API from Lab 5
  • Verify that no-token and valid-token calls behave correctly
  • Replace the hand-built resources with a Swagger import
  • Redeploy and smoke-test every method

Prerequisites 3 min

  • Labs 5a, 6a complete
  • Exported: $USER_ID, $ACCT, $API_ID, $ITEMS_ID, $LAMBDA_ARN, $URL, $POOL_ID, $ID_TOKEN

Starting fresh? bash ~/environment/dev-on-aws/bootstrap.sh 6b creates-or-reuses the API, Lambda, user pool + client, and captures a fresh $ID_TOKEN.

Step 1 — Create the Authorizer 8 min

AUTH_ID=$(aws apigateway create-authorizer \
  --rest-api-id $API_ID \
  --name cognito-$USER_ID \
  --type COGNITO_USER_POOLS \
  --identity-source method.request.header.Authorization \
  --provider-arns arn:aws:cognito-idp:us-east-1:$ACCT:userpool/$POOL_ID \
  --query id --output text)

echo "export AUTH_ID=$AUTH_ID" >> ~/.dev-on-aws.env
source ~/.dev-on-aws.env
echo "AUTH_ID=$AUTH_ID"

Persist AUTH_ID so Step 4's add_methods.sh can pick it up from ~/.dev-on-aws.env even in a new terminal.

Step 2 — Attach to POST 7 min

aws apigateway update-method \
  --rest-api-id $API_ID --resource-id $ITEMS_ID \
  --http-method POST \
  --patch-operations \
    op=replace,path=/authorizationType,value=COGNITO_USER_POOLS \
    op=replace,path=/authorizerId,value=$AUTH_ID 

aws apigateway create-deployment \
  --rest-api-id $API_ID --stage-name dev

Step 3 — Verify Auth 7 min

# No token → 401
curl -i -X POST $URL \
     -H "Content-Type: application/json" \
     -d '{"user":"u","title":"x","price":1}'

# Valid token → 200
curl -i -X POST $URL \
     -H "Authorization: $ID_TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"user":"alice","title":"authed","price":9.99}'

# Malformed token → 401
curl -i -X POST $URL -H "Authorization: not.a.token" \
     -d '{"user":"u","title":"x","price":1}'

Step 4 — Add GET & DELETE via Script 7 min

Open ~/environment/dev-on-aws/lab6/add_methods.sh in the editor to read what it does: adds GET on /items, creates the /items/{id} child resource, adds GET + DELETE there, deploys the stage. All protected by $AUTH_ID (from Step 1).

Terminal:

cd ~/environment/dev-on-aws/lab6
chmod +x add_methods.sh
./add_methods.sh
# Deployed.

Running a saved script (not pasting 25 lines into the terminal) is the reliable way to execute multi-line bash in Cloud9.

Step 5 — Smoke Test All Methods 7 min

H="-H Authorization:$ID_TOKEN"
curl -s $H $URL | jq .                           # GET list
curl -s $H -X POST $URL \
  -H "Content-Type: application/json" \
  -d '{"title":"from swagger","price":4.50}' | jq .   # POST
curl -s $H $URL/ITEM-123 | jq .                  # GET one
curl -s $H -X DELETE $URL/ITEM-123 -i             # DELETE

Success Criteria 3 min

  • ✅ Authorizer created and attached to POST
  • ✅ No-token → 401, valid-token → 200
  • ✅ All four methods (POST, GET /items, GET /items/{id}, DELETE /items/{id}) wired to Lambda with the Cognito authorizer
  • ✅ All four methods respond under the dev stage
